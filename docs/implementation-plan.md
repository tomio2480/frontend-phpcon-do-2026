# 北海道市区町村パーセンテージ可視化アプリ 実装計画

## 概要

GitHub Pages 上で動作する静的 Web アプリの実装計画である．
北海道の市区町村を地図とチェックボックスで選択し，
選択分の面積・人口・ふるさと納税受入額・受入件数が
北海道全体に対して占める割合を表示する．
結果を「あなたの北海道は何 % 」として X にシェアする機能も持つ．

集計ロジックはブラウザ上で動作する PHP WASM で実装する．
PHP カンファレンス北海道の登壇文脈と
ブラウザ単独動作の両立を狙う．

本ドキュメントは要件・技術選定・PHP と JS の責務分担・
データソース・アーキテクチャ・デザイン方針・
開発フェーズ・パフォーマンス対策・リスク・
アピールポイントを含む．

## 📋 要件整理

### 機能要件

選択操作は次の 2 系統を同期させる．

- 地図上のクリックによる市区町村の選択・解除
- チェックボックス一覧による選択・解除

選択結果に対し，次の値を即時に計算・表示する．

- 面積合計と全道に対する割合
- 人口合計と全道に対する割合
- ふるさと納税受入額合計と全道に対する割合
- ふるさと納税受入件数合計と全道に対する割合

すべての割合は小数点以下 2 桁固定で表示する．

シェア機能は次を満たす．

- 4 つの割合をまとめたテキストを生成する
- 何も選択されていない場合は割合をすべて 0.00 % として出力する
- X の Web Intent でポスト画面を開く
- URL は x.com 系の形式を採用する

### 非機能要件

- 静的サイトとして GitHub Pages 上で完結する
- ブラウザ単独で動作し，バックエンド API を要しない
- 初回ロード後の操作はオフラインでも動作させたい
- モバイルでも操作可能なレイアウトとする
- 集計ロジックは PHP で実装し WASM 経由でブラウザ実行する
- 各種ライブラリは原則として最新の安定版を採用する

## 🐘 PHP WASM 採用の背景

### 選定理由

PHP カンファレンス登壇の文脈にあるため，
ブラウザ上で PHP を動かす技術それ自体が題材になる．
WordPress Playground で実用化されている技術であり，
実験的なおもちゃではない．

集計のような軽量処理であれば PHP WASM のオーバーヘッドは
許容範囲に収まる．JS 単独実装と機能性は変わらないが，
「サーバを置かずに PHP のロジックを動かす」事例として価値がある．

### 採用するランタイム

WordPress Playground 系の `@php-wasm/web` を採用する．
直近の最適化でバイナリサイズが縮小し，
PHP バージョン単位でモジュールを分離できる構成になっている．

PHP のバージョンは 8.5 を採用する．
8.5 は 2025 年 11 月にリリースされた最新メジャーで，
`@php-wasm/web` でも公式サポート対象となっている．

### 採用しない選択肢

`seanmorris/php-wasm` は歴史ある実装で
`<script type="text/php">` も使えるが，
モジュール構成と更新頻度の観点で WordPress Playground 系を優先する．
サーバサイド PHP はそもそも静的サイト要件に反するので不採用．
PHP コードを事前にトランスパイルする方式は題材としての面白さが失われるため不採用．

## 🧩 PHP と JavaScript の責務分担

### 分担の原則

PHP WASM は計算・データ整形・テキスト生成に集中させ，
DOM 操作や地図の制御は JS 側に任せる．
境界をシンプルにすることで，
パフォーマンスと保守性の両方を確保する．

| 表 2 | 責務分担の整理 |

| 領域 | 担当 | 理由 |
| --- | --- | --- |
| 市町村データ保持 | PHP | 集計のたびに必要 |
| 選択集合の集計 | PHP | カンファレンス題材の中心 |
| 割合計算 | PHP | 上記と同居が自然 |
| X ポスト用テキスト生成 | PHP | 文字列整形は PHP 得意分野 |
| UI レンダリング | JS (Preact) | DOM 操作の責務 |
| 地図表示・操作 | JS (Leaflet) | DOM 操作の責務 |
| 状態管理 | JS | UI フレームワーク側で完結 |
| PHP との橋渡し | JS | ランタイムの API を扱う |

### 連携の流れ

JS で選択状態が変わるたびに，
市町村コード配列を PHP に渡して集計関数を呼び，
結果を JS に戻して UI に反映する．

```
[ユーザー操作] → [JS: 選択集合更新]
                     ↓
                [JS → PHP: 集計呼び出し]
                     ↓
                [PHP: 合計と割合を計算]
                     ↓
                [PHP → JS: 結果オブジェクト]
                     ↓
                [JS: UI 反映]
```

## 📊 データソース

### 基準日

データは 2026 年 5 月 27 日時点で取得できる最新のものを採用する．

### 面積データ

- 国土地理院「全国都道府県市区町村別面積調」
- 年 1 回更新
- 公開形式：Excel および PDF

### 人口データ

- 総務省統計局「住民基本台帳人口・世帯数」
- または北海道「住民基本台帳人口」
- 毎月更新．基準日を画面に明示する

### ふるさと納税データ

- 総務省「ふるさと納税に関する現況調査結果」
- 年 1 回更新（例年 7 〜 8 月公表）
- 受入額と受入件数の双方を取り込む

### 地理データ

- 国土数値情報「行政区域データ」
- 必要に応じて Shapefile から GeoJSON 変換
- mapshaper 等で 5 〜 10 % 程度に簡略化

### データ整形方針

データソースが Excel や CSV であるため，
事前に Python 等で JSON に整形してリポジトリに同梱する．
PHP WASM 上では JSON を読み込んで連想配列に展開する．
整形スクリプトは `scripts/` に置き，再現性を担保する．

札幌市の取り扱いは次節の通り「区を独立した行として保持」とする．
合計の二重計上を防ぐため，札幌市そのものはデータ行として持たない．

| ステップ | 入力 | 出力 |

2. Loading 表示

```ts

  population: number;

  furusatoCount: number;
```

## ⚙ 主要機能の実装方針

### PHP 側のコード例

集計関数と割合計算関数は単純な関数として書く．
グローバル変数や I/O は持たせず，
引数と戻り値で完結させる．

```php
<?php

declare(strict_types=1);

function calc_total(array $rows, array $codes): array
{
    $sum = [
        'area' => 0.0,
        'population' => 0,
        'furusato_amount' => 0,
        'furusato_count' => 0,
    ];
    foreach ($codes as $code) {
        $row = $rows[$code] ?? null;
        if ($row === null) {
            continue;
        }
        $sum['area'] += $row['area'];
        $sum['population'] += $row['population'];
        $sum['furusato_amount'] += $row['furusato_amount'];
        $sum['furusato_count'] += $row['furusato_count'];
    }
    return $sum;
}

function calc_percentages(array $sum, array $total): array
{
    $pct = function (float|int $part, float|int $whole): float {
        return $whole > 0 ? ($part / $whole) * 100.0 : 0.0;
    };
    return [
        'area_pct' => $pct($sum['area'], $total['area']),
        'population_pct' => $pct($sum['population'], $total['population']),
        'furusato_amount_pct' => $pct($sum['furusato_amount'], $total['furusato_amount']),
        'furusato_count_pct' => $pct($sum['furusato_count'], $total['furusato_count']),
    ];
}
```

選択が空でも `calc_total` は 0 を返し，
`calc_percentages` も 0 を返す．
ポスト文言の組み立てもこの結果を素直に使えばよい．

### JS からの呼び出し方針

PHP WASM のランタイムに JSON 文字列を渡して受け取り，
パース後に UI に反映する．
データは仮想ファイルシステムに置き，
PHP の関数呼び出しのみで集計を完結させる構成を本線とする．

```ts
const payload = JSON.stringify(Array.from(selection));
const result = await php.runStream({
  code: `<?php
    require '/app/aggregate.php';
    $rows = json_decode(file_get_contents('/data/municipalities.json'), true);
    $total = json_decode(file_get_contents('/data/total.json'), true);
    $codes = json_decode(${JSON.stringify(payload)}, true);
    $sum = calc_total($rows, $codes);
    $pct = calc_percentages($sum, $total);
    echo json_encode($sum + $pct);
  `,
});
const aggregated = JSON.parse(await result.stdoutText);
```

### X へのポスト

ポスト用テキストの生成は PHP 側で行う．
桁数を 2 桁固定にし，
何も選択されていない場合は 0.00 % が並ぶ．

```php
function build_post_text(array $pct, string $url): string
{
    return sprintf(
        "あなたの北海道は…\n面積 %.2f %%\n人口 %.2f %%\nふるさと納税(額) %.2f %%\nふるさと納税(件数) %.2f %%\n#あなたの北海道は何パーセント\n%s",
        $pct['area_pct'],
        $pct['population_pct'],
        $pct['furusato_amount_pct'],
        $pct['furusato_count_pct'],
        $url
    );
}
```

JS 側は PHP から受け取ったテキストを Web Intent URL に乗せる．
URL は x.com 系の現行形式を使う．

```ts
const url = new URL('https://x.com/intent/post');
url.searchParams.set('text', postText);
window.open(url.toString(), '_blank', 'noopener,noreferrer');
```

ハッシュタグは本文中に含める方式とし，
`hashtags` パラメタは併用しない．
これによりタグの順序や位置を PHP 側で完全に制御できる．

## 🚀 パフォーマンス対策

### 初回ロード対策

PHP WASM の初期化には数 MB の通信と
数百ミリ秒〜数秒の処理が必要となる．次の対策を組み合わせる．

- Service Worker による `.wasm` のキャッシュ
- HTTP キャッシュヘッダ任せにせず明示的にバージョン管理
- Loading オーバーレイで体感待ち時間を緩和
- `<link rel="preload">` で WASM の取得を前倒し
- 必要な PHP モジュールだけを含むビルドを選ぶ

### 二回目以降のロード対策

Service Worker でランタイムをキャッシュすれば，
オフラインでも起動する．
キャッシュ更新時にバージョン文字列を付け替え，
古い WASM を破棄する．

### 集計呼び出しの軽量化

選択変更のたびに PHP に問い合わせると
応答性に影響する場合がある．次のいずれかで対処する．

- 変更を 50 〜 100 ms デバウンスする
- データを PHP メモリに事前展開し関数呼び出しのみで完結させる
- Web Worker 内で PHP WASM を動かし UI スレッドを止めない

### モバイルでのメモリ

スマートフォンの低メモリ機種ではタブが落ちる可能性がある．
データ JSON のサイズと WASM ヒープを抑える．
不要な PHP 拡張は無効化する．

## 🚀 GitHub Pages デプロイ

### Vite 設定

リポジトリ名をパスに含むためベースパスを設定する．

```ts
// vite.config.ts
export default defineConfig({
  base: '/hokkaido-percentage/',
  plugins: [preact()],
  assetsInclude: ['**/*.wasm', '**/*.php'],
});
```

### WASM の MIME type

GitHub Pages は `.wasm` を `application/wasm` として返す．
問題なくストリーミングコンパイルが利用できる．
設定変更は不要だが，初回デプロイ後に DevTools でレスポンスヘッダを確認する．

### COOP / COEP の制約

`SharedArrayBuffer` を必要とする機能（マルチスレッド WASM 等）は，
COOP/COEP ヘッダの設定が必要となる．
GitHub Pages はカスタムヘッダを設定できないため，
マルチスレッド機能には依存しない構成を選ぶ．
`@php-wasm/web` は単一スレッド構成で動作可能であり，
本アプリではそちらを採用する．

### GitHub Actions ワークフロー

`actions/deploy-pages` を用いる．権限は最小に絞る．

```yaml
permissions:
  contents: read
  pages: write
  id-token: write
```

`main` への push 時にビルドして Pages にデプロイする．
初回の手動有効化（ Settings → Pages → Source: GitHub Actions ）が必要．

### push の取り扱い

ユーザー指示の通り， `git push` は明示的な指示があるまで行わない．
ローカルでビルドして `npm run preview` で動作確認した上で，
push 可否を判断する手順を踏む．

## ⚠ リスクと検討事項

### 初回ロード待ち時間

PHP WASM 自体が数 MB あり，
4G 回線では数秒の追加待機が発生する可能性がある．
Loading 表示と Service Worker キャッシュで二回目以降を高速化する．
それでも快適とは言えない初回体験を許容するかは要判断．

### PHP 8.5 の安定性

PHP 8.5 はリリースから約半年が経過した最新メジャーである．
`@php-wasm/web` 側のサポートも入っているが，
バグ報告のチャネルを開発期間中に注視する．
重大な問題があれば 8.4 へのフォールバックを検討する．

### JS 実装との二重実装リスク

集計ロジックを PHP に置くと，
仮に PHP WASM が動かない環境向けの JS 版フォールバックを書きたくなる．
二重実装は保守負荷が高いため，
本アプリでは PHP WASM 単一構成とし，
対応ブラウザを明示することで割り切る．

### GeoJSON のサイズ

行政区域データは無加工だと数 MB に達する．
mapshaper 等で 5 〜 10 % まで簡略化し，
初回ロード時間を抑える．

### 市区町村コード変動の扱い

北海道では直近の合併・廃止は無いが，
将来の変動に備え，コードと名称の対応表を別管理にする．

### モバイルでの動作

スマートフォンで地図クリックと
スクロール・ピンチ操作の競合が起きやすい．
加えて PHP WASM のメモリ使用量がモバイルで問題になりうる．
低スペック機での挙動を実機で確認する．

### iOS Safari の挙動

iOS Safari は WASM 関連の挙動が他ブラウザと異なるケースがある．
WordPress Playground でも対応実績はあるが，
本アプリでも早期に検証する．

### キャッシュとデータ更新

ユーザーがブラウザにキャッシュされた旧データを見続ける可能性がある．
ファイル名にバージョンや日付を付け，
更新を確実に反映できる仕組みを入れる．
Service Worker の更新フローも設計する．

### Leaflet の長期未更新

Leaflet 1.9.4 は 2023 年 5 月以降更新されていない．
セキュリティ脆弱性や代替ライブラリへの移行を
定期的に確認する運用が必要．
